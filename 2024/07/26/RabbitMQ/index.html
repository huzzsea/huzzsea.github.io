第一节 RabbitMQ基本概念

1.1 消息队列

  • 消息队列：实现应用程序和应用程序之间通信的中间件产品

  • 消息队列的功能

    • 功能解耦
    • 快速响应
    • 削峰限流
    • 便于扩展

  • 底层通信标准

    • AMQP：通用协议，IBM 公司研发
    • JMS：专门为 Java 语言服务，SUN 公司研发，一组由 Java 接口组成的 Java 标准

  • 底层通信标准对比

    • 七层网络模型
      • AMQP 对应传输层与会话层
      • JMS 对应应用层
    • 消息模型
      • AMQP 支持多种消息模型，包括点对点和发布/订阅
      • JMS 主要支持点对点和发布/订阅两种消息模型
    • 支持的编程语言和平台
      • AMQP 支持多种编程语言和平台，包括 Java、C++、Python 等
      • JMS 主要针对 Java 平台，因此在其他编程语言和平台上的支持相对较少
    • 可靠性
      • AMQP 提供了强大的消息可靠性保证，包括消息持久化、事务性消息和消息确认机制
      • JMS 也支持消息持久化和事务性消息，但具体实现取决于消息传递系统的提供者
    • 传输协议
      • AMQP 使用二进制协议进行消息传递，提供了高效、可靠的消息投递机制
      • JMS 使用一种面向文本的协议，消息的传输效率可能较低
    • 扩展性和兼容性
      • AMQP 具有很好的拓展性和兼容性，可以在不同的消息代理之间交互操作
      • JMS 在 Java 环境中有较好的拓展性和兼容性，但在与非 Java 环境集成时受到限制

1.2 RabbitMQ概述

  • RabbitMQ：基于 AMQP、由 Erlang 语言开发的消息队列产品

  • RabbitMQ体系结构

    • Producer：消息的发送端，生产者
    • Consumer：消息的接收端，消费者
    • Connection：生产者到消息队列主体服务器之间的TCP连接
    • Channel：在一个已经建立的 Connection 中建立的逻辑连接，每个 Channel 都有自己的 id，实现 Connection 复用
    • Broker：主体服务器本身，负责接受消息和分发消息
    • Virtual Host：每一个 Virtual Host 就是一个虚拟分组，用户在自己的 Virtual Host 中使用 RabbitMQ 组件
    • Enchange：交换机，是消息达到 Broker 的第一站
    • Queue：队列，是消息的容器，消息放在这里等待被消费端取走

1.3 RabbitMQ工作模式

  • Work Queues
    • 多个消费者监听同一个队列，各消费者之间对同一个消息是竞争的关系
    • 适用于任务较重或任务较多的情况，多消费者分摊任务可以提高消息处理的效率
  • Publish/Subscribe
    • 生产者把消息发送到交换机
    • 交换机有如下3种常见类型
      • Fanout：广播，将消息发送给所有绑定到交换机的队列
      • Direct：定向，把消息交给符合指定路由键的队列
      • Topic：通配符，把消息交给符合路由模式的队列
    • 队列直接和交换机绑定
  • Routing
    • 通过路由绑定的方式，把交换机和队列关联起来
    • 交换机和队列通过路由键进行绑定
    • 生产者发送消息时不仅要指定交换机，还要指定路由键
    • 交换机接收到消息会发送到路由键绑定的队列
  • Topics
    • 让队列在绑定路由键的时候使用通配符
  • RPC
    • 远程过程调用，本质上是同步调用

第二节 RabbitMQ机制

2.1 消息可靠性投递

  • 故障情况

    • 消息没有发送到消息队列上
    • 消息队列服务器宕机
    • 消费端出现问题

  • 解决方案

    • 消息没有发送到消息队列

      • 在生产者端进行确认，分别针对交换机和队列来确认，如果没有成功发送到消息队列服务器上，那就尝试重新发送

        • 编写配置

        • 配置类

      • 为目标交换机指定备份交换机，当目标交换机投递失败时，把消息投递至备份交换机

        • 创建备份交换机，类型为 fanout
        • 创建队列，绑定交换机

    • 消息队列服务器宕机

      • 消息持久化到硬盘上，哪怕服务器重启也不会导致消息丢失
        • 在后台管理界面创建交换机和队列时，默认就是持久化的模式

    • 消费端出现问题

      • 消费端消费消息成功，给服务器返回ACK信息，然后消息队列删除该消息，否则给服务器端返回NACK信息，同时把消息恢复为待消费的状态

        • 编写配置

        • 消费者监听器

2.4 死信和死信队列

  • 死信：当一个消息无法被消费，它就变成了死信
  • 死信的来源
    • 拒绝：消费者拒接消息，并且不把消息重新放入原目标队列
    • 溢出：队列中消息数量到达限制
    • 超时：消息到达超时时间未被消费
  • 死信的处理方式
    • 丢弃：对不重要的消息直接丢弃，不做处理
    • 入库：把死信写入数据库，日后处理
    • 监听：消息变成死信后进入死信队列，我们专门设置消费端监听死信队列，做后续处理
  • 死信处理方式
    • 创建死信交换机
    • 创建死信队列
    • 消费者监听死信

2.6 事务消息

  • 事务消息

    • 在生产者端使用事务消息和消费端没有关系
    • 在生产者端使用事务消息仅仅是控制事务内的消息是否发送
    • 提交事务就把事务内所有消息都发送到交换机
    • 回滚事务则事务内任何消息都不会被发送

2.7 惰性队列

  • 惰性队列：以惰性模式运行的经典队列，队列中的消息会尽可能早地移动到磁盘，这些消息仅在消费者请求时加载到 RAM 中

  • 应用场景

    • 消费者离线/崩溃/停机进行维护
    • 突然出现消息进入高峰，生产者的速度超过了消费者
    • 消费者比正常情况慢

  • 设置惰性队列
